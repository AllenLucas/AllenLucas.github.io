普通类型Dialog对应的Context为什么必须是Activity

发表于 2021-07-05 | 分类于 Android Dialog | | 本文总阅读量 次

文章截选自鸿洋大神的文章

问题

  1. 为什么传入一个非Activity的context会出现错误？
  2. 传入的context一定要Activity吗？

解析

  1. 先来看第二问，这里只说结论，普通的Dialog传入的Activty必须使用Activity，但是如果指定为系统级的dialog，是可以传入非Activity的context的。那么为什么会这样呢。接下来一点一点的详解：

    • 先说一下如果指定为系统级的dialog，

      • 首先，涉及到系统类型的为防止滥用，肯定会授予权限才可以，这里必须要拥有SYSTEM_ALERT_WINDOW权限才可以
      • 在调用Dialog.show方法之前，必须把Dialog的Window的type指定为SYSTEM_WINDOW类型，比如TYPE_SYSTEM_ALERT or TYPE_APPLICATION_OVERLAY 这样的类型

    • 普通的Dialog的容器为Activity，所以其窗口属性的token就是引用的Activity中的token，在WMS中，会根据该token来查找到对应的ActivityRecord实例，后面会把Dialog对应的WindowState添加到该ActivityRecord中。note:如果在查找容器这一步，没有找到对应的实例，就会抛出BadTokenException(token null is not valid:is your activity running?)

    • 为什么使用Activity才可以找到容器，其他的非Activity的Context不行呢？Dialog在show方法中是通过WindowManager来添加view的，而这个WindowManager对象是从Context的getSystemService(WINDOW_SERVICE)方法获取的

      重点,与其他非Activity不同，Activity重写了context的getSystemService方法，在获取WINDOW_SERVICE的时候返回了Activity主Window的WindowManager对象，其中的mParentWindow指向主Window(其他非Activity的Context的WindowManager.mParaentWindow默认为null)

      WindowManagerGlobal在addView的时候，如果检查到mParentWindow不为null，就会对窗口属性的token进行赋值，逻辑为

      • 窗口类型为SUB_WINDOW(子窗口),会把mParentWindow对应的ViewRootImp的mWindow赋值给token
      • 窗口类型为SYSTEM_WINDOW(系统级别窗口)，不会对token进行赋值，因为普通应用的Window等级低于系统Window
      • 窗口类型APPLICATION_WINDOW,会把mParentWindow的mAppToken(所属Activity的mToken)赋值给token

    • 为什么需要一个Token，WMS需要根据该token确定Window的位置，如果没有token无法确认放在哪个容器上。

    • 为什么设置为系统级别弹窗，没有token也可以显示呢？当获得SYSTEM_ALERT_WINDOW权限后，会即时创建一个WindowToken，然后新建的WindowToken附加到特定的容器上.

    右边地下的WindowState就是系统级别类型dialog的窗口，层级关系上，与隔壁的ActivityRecord相等。这里的WindowToken就是刚刚即时创建的WindowToken

  2. 回到第一个问题，为什么使用非Activity来创建并弹出dialog，有时会发生BadTokenException？

    • 因为使用非Activity来创建，无法获取到parentWindow，导致在WMS里找不到对应的容器
    • 另一个就是想要使用系统级dialog，但是没有SYSTEM_ALERT_WINDOW权限，同样当作普通dialog来处理

    两种情况都是无法找到对应的盛放容器，所以报错

总结

show一个普通的Dialog需要的并不是Activity本身，而是一个容器的token，平时传Activity，只是刚好有对应WMS的一个WindowState容器而已
